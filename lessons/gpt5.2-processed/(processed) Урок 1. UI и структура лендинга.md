## Что такое лендинг и зачем он нужен

**Лендинг (landing page)** — это одностраничный сайт, у которого есть одна главная цель: привести пользователя к целевому действию. Чаще всего это **оставить контакты** (лид), чтобы вы могли продолжить коммуникацию: позвонить, написать, сегментировать, догреть.

Важно: лендинг не должен «рассказывать всё обо всём». Он должен **вести к действию** — шаг за шагом, снимая вопросы и сомнения.

---

## Из чего состоит лендинг

Есть пять обязательных секций, которые дают странице понятную логику:

* **Hero** — кому и зачем этот продукт
* **Proof** — почему нам можно доверять
* **Benefits** — какую конкретную пользу получит пользователь
* **FAQ** — снятие возможных сомнений и возражений
* **CTA** — призыв к действию, ради которого и создавался лендинг

Это не просто «красивые блоки». Это маршрут пользователя: от первого впечатления → к доверию → к выгодам → к ответам на сомнения → к действию.

---

## Разбираем секции по смыслу

<пример Hero>

### Hero

Это первый экран, который видит пользователь.
Задача Hero — за несколько секунд объяснить «что это» и «почему мне это может быть нужно».

Обычно здесь размещают:

* **основной заголовок**, чтобы сразу обозначить общий контекст;
* **короткое пояснение**, для кого продукт и зачем он — чтобы заинтересовать и побудить к скроллу;
* **кнопку/ссылку**, ведущую к CTA-блоку — чтобы можно было сразу перейти к действию.

<пример Proof>

### Proof

Если человек не доверяет — он не оставит контакты, даже если ему интересно.

Поэтому здесь уместны:

* конкретные **факты и цифры** (не «лучшие на рынке», а измеримое);
* **примеры кейсов** (что было → что стало);
* **клиенты/компании**, которые уже пользовались продуктом.

<пример Benefits>

### Benefits

Это самая «продающая» часть страницы: здесь пользователь должен понять, **что именно он получит**.

Чаще всего блок выглядит так:

* **3–6 конкретных выгод**;
* короткое пояснение к каждой;
* подача «меньше текста — больше ясности»: визуальные акценты, короткие формулировки.

<пример FAQ>

### FAQ

Если пользователь дошёл до FAQ — он уже почти готов, но всё ещё сомневается.
Задача FAQ — проговорить страхи и уменьшить барьер.

Какие вопросы можно включить:

* “Нужен ли опыт?”
* “Сколько времени это занимает?”
* “Есть ли скрытые условия?”
* “Что если мне не подойдёт?”

<пример CTA>

### CTA

Это финальная секция: здесь человек принимает решение.

Чтобы помочь ему сделать шаг, в CTA обычно есть:

* короткий заголовок;
* повтор ценности (1–2 предложения);
* форма или кнопка;
* **минимально необходимое количество полей**, чтобы не «убить» конверсию.

## Практика: собираем лендинг своими руками

А теперь применим знания и сделаем собственный лендинг.

План простой:

* установить на компьютер IDE Visual Studio Code <ссылка на отдельный урок>
* настроить нужные плагины для взаимодействия с выбранной LLM
* составить запросы на создание лендинга и передать их в LLM
* после чего запустить результат в браузере

Для примера создадим лендинг предварительной регистрации пользователей на будущий курс Яндекс.Практикум «Вайбкодинг».

Открываем окно запроса AI агента GPT-5.3-Codex и отправляем ему запрос:

```
Ты профессиональный интернет-маркетолог с навыками frontend-разработчика.  

Необходимо разработать лендинг предварительной регистрации пользователей на обучающий курс для маркетологов, проджект и продакт менеджеров под названием "Вайбкодинг".

Требования:
- лендинг должен состоять из блоков: Hero, Proof, Benefits, FAQ, CTA
- Hero содержит заголовок и призыв к действию в блоке CTA
- Proof содержит рандомные продающие показатели и кейсы реальных покупателей курса. Оставь плейсхолдеры под фото клиентов
- Benefits - придумай 3 подходящих под контекст выгод прохождения курса для потенциального студента с кратким пояснением по каждому
- FAQ - придумай 3 наиболее вероятных вопроса покупателя курса и подходящие ответы на них
- CTA должен содеражать форму отправки лида с полями: имя (строка), телефон (строка), чекбокс согласия обработки персональных данных и кнопку отправки формы с подходящим текстом
  
Дополнительные инструкции:
- каждый блок имеет якорь для навигации
- для написания лендинга использовать только: html, css
- результат работы разместить в файле index.html
- верстка страницы адаптивная
  
Работу выполняй поэтапно:
- разметь основную структуру страницы
- последовательно заполняй блоки
- соблюдай логическую связность между блоками - каждый последующий блок это борьба с потенциальными возражениями клиента и усилия на перевод клиента на CTA форму
```

После ответа агента в папке проекта появится файл **index.html**.
Открываем его двойным кликом в браузере — и смотрим первый результат.

## Следующий шаг: делаем из вёрстки «живой» инструмент

Сама по себе страница — это только «витрина». Чтобы она начала приносить пользу в процессах, нужно уметь **принимать данные из формы** и дальше с ними работать.

Для этого пишем агенту:

```
Лендинг index.html нужно преобразовать во frontend-приложение, которое будет локально запускаться в docker-контейнере.

Необходимо:
- создать локальный git репозиторий
- создать коммит текущего index.html лендинга
- описать docker-compose файл с контейнером app: node.js, express
- преобразовать лендинг в index.html во frontend приложение на typescript
- создать README.md с инструкцией запуска fronted-приложения с лендингом
```

После того как агент завершит работу:

1. открываем **README.md**,
2. выполняем инструкции,
3. запускаем лендинг как полноценное frontend-приложение — дальше будем добавлять к нему нужные функции.
